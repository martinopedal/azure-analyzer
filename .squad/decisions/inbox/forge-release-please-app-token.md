# Decision: release-please switched from GITHUB_TOKEN to GitHub App token

**Date:** 2026-04-24
**Author:** Forge
**PR:** #955
**Issue:** #954

## Context

Release-please uses `googleapis/release-please-action` to create version-bump PRs on push to `main`. When configured with `GITHUB_TOKEN`, PRs it creates do not trigger downstream CI workflows due to GitHub's anti-recursion guard. This left release PR #932 in BLOCKED state with zero CI checks.

## Decision

Switch the `release_please` job's token source from `${{ secrets.GITHUB_TOKEN }}` to a GitHub App token generated by `actions/create-github-app-token@v2` (SHA-pinned).

### Why GitHub App over PAT?

| Factor | PAT | GitHub App |
|--------|-----|------------|
| Tied to user | Yes | No |
| Token lifetime | Up to 1 year | 1 hour (auto-rotated) |
| Permission scope | Broad | Scoped per-installation |
| Audit trail | User identity | App identity |
| Rotation | Manual | Automatic |

## Implementation

- Added `actions/create-github-app-token@fee1f7d63c2ff003460e3d139729b119787bc349 # v2` step before release-please
- Release-please step now uses `${{ steps.app-token.outputs.token }}`
- App requires `contents: write` + `pull-requests: write` only
- Credentials: `RELEASE_APP_ID` + `RELEASE_APP_PRIVATE_KEY` repo secrets
- Only `release_please` job modified; `publish` job untouched
