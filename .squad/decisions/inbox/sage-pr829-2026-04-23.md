# Decisions inbox — Sage, PR #829 (2026-04-23)

## Context
PR #829 was a post-cascade docs refresh authored by copilot-swe-agent for issue #827. Picked up in BEHIND state with 7 unresolved Copilot review threads.

## Decisions

1. **Merge (not rebase) origin/main into the agent branch.** Preserves the agent's commit topology so the "Cloud agent PR review contract" still maps threads to original SHAs. Agent branches are routinely squash-merged to main, so the extra merge commit is discarded on squash — no history cost.

2. **Hand-fix sample reports, don't regenerate.** `samples/sample-report.md|html|mockup.md` are static sample artifacts, not Generate-ToolCatalog.ps1 output. They must stay internally consistent between MD and HTML. The Copilot threads flagged three real inconsistencies in the refreshed samples: 36 vs 37 tool count, D vs F posture grade, and `finops-signals` vs `finops` tool id. Fixed all three to the HTML / manifest-truthful value.

3. **Replace hardcoded catalog counts with manifest-driven prose.** `docs/reference/README.md` previously said "currently 36 enabled + 1 opt-in" — this drifts whenever tool-manifest.json changes. Rephrased to "Enabled + opt-in tools from the manifest (counts reflect the current manifest)." so the file stays correct without needing catalog-regen cycles.

4. **Accept the HTML sample tool-versions table being hand-populated.** The cleaner fix would be to make the HTML report generator always emit the table. Out of scope for a docs PR. Filed as future debt — see follow-up suggestion below.

5. **Treat `em-dash policy` HTTP 500 on git fetch as transient.** Log showed 3x HTTP 500 from `github.com` during checkout, not a policy violation. `gh run rerun --failed` cleared it. Not a real em-dash regression.

## Follow-up suggestions (not filed as issues)

- **Generator for sample reports.** The three samples (`sample-report.md`, `sample-report.html`, `sample-report-v2-mockup.md`) drift easily because they're hand-maintained. Consider a `scripts/Generate-SampleReports.ps1` that renders the same fixture JSON through `New-MdReport` / `New-HtmlReport` / mockup template, then commit the output. Would have prevented this entire review cycle.
- **Manifest count autotext.** `docs/reference/README.md` could embed a `<!-- manifest-counts -->` marker that `Generate-ToolCatalog.ps1` auto-fills, same pattern as README badges. Would let us keep the exact numbers without drift risk.
- **CRLF handling in Generate-ToolCatalog.ps1.** Windows runs produce LF-output that git converts to CRLF, yielding phantom diffs. Consider setting `[System.IO.File]::WriteAllText` with the platform newline to avoid false positives on `git status` during local regen.
