# Forge decision: auto-managed doc coverage (PR `chore/auto-doc-coverage`)

## Context

Auto-coverage matrix audit (Martin, 2026-05-12) flagged three "human-touched gaps" in the
docs pipeline:

| Doc | Gap | Risk |
|---|---|---|
| `PERMISSIONS.md` INDEX | drift-checked but not regenerated by the bumper | every pin bump leaves PERMISSIONS stale until a human re-runs the generator |
| `docs/reference/permissions/<tool>.md` per-tool pages | generator THREW on missing page | adding a tool to the manifest blocked the bumper hard |
| `README.md` tool count + supported list | hard-coded "36 read-only assessment tools (+ 1 opt-in)" | drift on every add/remove; was already stale (manifest exposes 38) |

We fix all three in one PR so the bumper produces a single atomic commit per tool bump
with the manifest + catalogs + PERMISSIONS + README all in lockstep.

## Decisions

### D1. Sibling generator over fat-script extension (Fix 3)

For the README facts I considered two options:

- **(a) Extend `scripts/Generate-ToolCatalog.ps1`** so a single generator owns both the
  tool-catalog files AND the README facts.
- **(b) Create `scripts/Generate-ReadmeFacts.ps1` as a sibling generator** mirroring the
  shape of `Generate-PermissionsIndex.ps1`.

I picked **(b)** for four reasons:

1. **Single Responsibility**. `Generate-ToolCatalog.ps1` projects the manifest into two
   reference Markdown files (`docs/reference/tool-catalog{,-contributor}.md`). Mixing in
   `README.md` patching would make the script own two unrelated surfaces with two
   different marker conventions.
2. **Test isolation**. The existing `tests/scripts/Generate-ToolCatalog.Tests.ps1`
   contract checks that running the generator produces exactly the catalog files (and
   nothing else). Bolting on README writes would force a rewrite of that contract.
3. **Per-invariant CI gate**. The docs-check workflow now has three named jobs
   (`tool-catalog-fresh`, `permissions-pages-fresh`, `readme-facts-fresh`) with
   one-line job descriptions on the PR check list. Folding README facts into
   tool-catalog would muddy that signal.
4. **Bumper invocation pattern is already per-script**. `Update-ToolPins.ps1` already
   shells out to one `pwsh -File scripts/Generate-*.ps1` per generator, so a sibling
   script costs one extra invocation block (with its own try/catch and Write-Warning
   diagnostic) and zero refactor.

### D2. H1 stub format = `{displayName} - Required Permissions` (deviates from the spec)

The task spec called for `# {displayName} — Permissions`. I deviated on two points:

- **Use ASCII hyphen, not em dash.** Em dashes are banned across docs/markdown by the
  PR template + markdown-check. The existing `Generate-ToolCatalog.Tests.ps1` test
  `neither catalog contains an em dash` enforces the same on the catalog generators,
  so I extended the convention to the new stub generator.
- **End the H1 with `- Required Permissions`.** The pre-existing per-tool page contract
  test `tests/scripts/Generate-PermissionsIndex.Tests.ps1::every per-tool page has the
  standard H1 heading` asserts `^# .* - Required Permissions$`. Stubbing with
  `# {displayName} - Permissions` would have introduced a new file that fails the
  contract on its first commit. I kept the stub aligned with the existing convention.

### D3. Mirror stubs across the consumer/reference directory drift (legacy bug)

`Generate-PermissionsIndex.ps1` checks for missing pages in `docs/reference/permissions/`
(default `$PagesDir`) but emits index links pointing at `docs/consumer/permissions/`.
Both directories exist today and contain the same 45 files (38 enabled tools + 5
cross-cutting + README). This is pre-existing repo drift introduced when #418 added
the second directory without a follow-up consolidation.

I did NOT consolidate the directories in this PR (out of scope, would touch every link
in the rendered index). Instead the auto-stub helper writes the stub to BOTH directories
(when both exist) so the rendered index links resolve and both `permissions-pages-fresh`
candidates are satisfied.

The directory consolidation deserves its own PR. Filed as a follow-up note in
forge/history.md.

### D4. Static "+ 1 opt-in" literal in the README facts generator

The "+ 1 opt-in" marketing label refers to `copilot-triage` (the only currently-shipped
opt-in tool, gated by `-EnableAiTriage`). The other 13 disabled manifest entries are
pre-registered scaffolding for follow-up R2..R4 PRs (EASM family, graph mapping family)
that are not user-runnable yet, so they should not count.

For now I hardcode the opt-in count to `1` with a code comment pointing at the future
extension path: when a second shipped opt-in lands, switch the count to a manifest
projection over `comment -match 'opt-in'` (or introduce an explicit `optIn: true`
schema flag).

### D5. Did NOT auto-generate `THIRD_PARTY_NOTICES.md`

The audit asked whether `THIRD_PARTY_NOTICES.md` should mirror the manifest. Inspection
shows it is a hand-curated SUBSET (11 entries) focused on copyright + license attribution
for the legally significant upstream dependencies. It is not a 1:1 manifest projection
(skips first-party tools, skips PowerShell modules with no attribution requirements,
adds first-party components in a separate section). Auto-generation would be wrong here
since copyright / license / install hints cannot be inferred from the manifest schema.

Outcome: leave `THIRD_PARTY_NOTICES.md` hand-curated. Documented this rationale in the
PR body. No new generator, no new CI gate, no marker pairs.

## Verification

- `Invoke-Pester -Path .\tests\scripts -Output Detailed` -> 86 passed, 0 failed (3 new
  contexts added).
- Full repo `Invoke-Pester -Path .\tests` -> 3071 total / 3022 passed / 1 failed
  (pre-existing live-tool gitleaks transient, unrelated) / 48 skipped. Baseline floor
  (`$MinTotal=1680`, `$MinPassed=1645`) preserved with massive headroom; no floor bump
  needed.
- `pwsh -File scripts/Generate-ReadmeFacts.ps1 -CheckOnly` -> exit 0 on the regenerated
  README.
- `pwsh -File scripts/Generate-PermissionsIndex.ps1 -CheckOnly` -> exit 0 on the
  regenerated PERMISSIONS index.
