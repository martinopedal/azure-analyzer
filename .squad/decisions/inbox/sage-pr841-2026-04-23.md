## Summary
LiveTool wrapper smoke was failing because `Invoke-Gitleaks` and `Invoke-Trivy` could emit `Findings = $null` on success paths, while the contract (and every consumer of the v1 envelope + v2 normalizer) expects `Findings` to be an array under every path. PR #841 makes the contract explicit in code and test, and also fixes the LiveTool scorecard install probe (`scorecard --version` → `scorecard version`).

## Changes landed
- `modules/Invoke-Gitleaks.ps1` — success return now wraps with `Findings = @($findings)` so a single object never collapses and an empty result never appears as `$null`.
- `modules/Invoke-Trivy.ps1` — same non-null array contract on success.
- `tests/wrappers/Invoke-Gitleaks.Tests.ps1` / `tests/wrappers/Invoke-Trivy.Tests.ps1` — regression assertions combining `PSObject.Properties['Findings']` non-null check with the comma-operator `,$result.Findings | Should -Not -Be $null` plus `@($result.Findings).Count | Should -Be 0`. The comma operator is the key — without it an empty `@()` collapses and the `Not -Be $null` assertion misleads.
- `.github/workflows/ci.yml` — LiveTool probe uses `scorecard version`; added `# tracked: martinopedal/azure-analyzer#604` markers above the two step-level `continue-on-error: true` blocks that appeared during conflict resolution with #867.
