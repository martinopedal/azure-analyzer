## What this PR delivers

1. **`docs/audits/track-d-tool-fidelity.md`** — a 36x12 coverage matrix and a
   vendor-specific enrichment table. Identifies exactly which schema 2.2
   additive fields are dropped per normalizer.
2. **3 enrichment patches** — surgical edits to the three normalizers with the
   worst fidelity loss (PSRule, Kubescape, Scorecard). Each patch is purely
   additive: derives values when the wrapper omitted them, passes through
   wrapper-supplied values unchanged, and never mutates existing field plumbing.
3. **`tests/normalizers/Track-D-Fidelity.Tests.ps1`** — fixture-based
   regression tests that pin every enriched field with an explicit
   assertion, so any future regression that drops a field is caught at CI.
4. **CHANGELOG entry** under `[Unreleased] -> Added`.

## Audit findings

### Coverage matrix summary

37 wrappers are registered in `tools/tool-manifest.json` (one — `copilot-triage`
— is intentionally collector-only with no normalizer). Of the 36 normalizers:

- **2** carry all 12 schema 2.2 additive fields (AzGovViz, Falco)
- **6** drop a single dimension (most commonly `ScoreDelta`)
- **20** drop 2-4 dimensions (`MitreTactics/Techniques` are by far the most common)
- **8** drop 5-7 dimensions — these are the priority backlog

### Top-3 worst fidelity loss

| Rank | Normalizer | Dimensions dropped                   | Vendor cost                                                                  |
| ---: | ---------- | ------------------------------------ | ---------------------------------------------------------------------------- |
|    1 | PSRule     | 7 (Imp, Eff, EU, SD, MTac, MTec, ER) | Largest Azure-side finding volume; UX heatmap blank for impact/effort        |
|    2 | Kubescape  | 6 (Imp, Eff, DL, RS, SD, ER)         | AKS cluster cards have no fix-it deep links and no inline remediation snippets |
|    3 | Scorecard  | 6 (Imp, Eff, SD, MTac, MTec, ER)     | OpenSSF score (0-10) never surfaces as a delta in trend graphs               |

### Vendor-specific fields currently lost in `entities.json`

The single highest-cost regression is **Defender for Cloud's
`recommendation.metadata.{userImpact,implementationEffort}`** which would
populate `Impact` / `Effort` end-to-end without any inference. Patch deferred
to #432c (Defender already lands 9 of 12 dims so it does not enter the top-3).

Maester's `severityScore` (numeric 0-100) similarly maps cleanly to
`ScoreDelta` and is filed for #432c.

Trivy's `cve.cvssV3` / `fixedVersion` are folded into prose `Detail` /
`Remediation` — surfacing them as structured fields requires a wrapper-side
change (parse the SARIF) and is therefore filed against the **wrapper**, not
the normalizer.

## What is intentionally NOT in this PR

- **Wave-2 patches** for KubeBench, Sentinel*, Prowler, WARA, etc. The audit
  doc lists them in section 5 as `#432c` candidates.
- **Wrapper-side changes** for Trivy/Defender/Maester to expose new structured
  fields. Filed as separate audit follow-ups.
- **Renderer changes**. Track F (#434) consumes the new fields; this PR only
  guarantees they are present in `entities.json`.

## Risk assessment

- **Schema risk:** zero. All edits use existing `New-FindingRow` parameters,
  no new fields are introduced.
- **Test risk:** existing per-normalizer test files are untouched; only a new
  test file is added so no baseline assertion shifts.
- **Sanitisation risk:** zero. The patches do not introduce any new file
  output; everything routes through `New-FindingRow` which is already
  sanitised by the v2 schema layer.
- **Performance risk:** negligible. Each patch adds O(1) string assignments
  per finding.
