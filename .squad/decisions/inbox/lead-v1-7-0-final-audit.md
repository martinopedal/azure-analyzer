## PART 2: Release-Please Policy Investigation + Recommendation

### The Problem
Today's `chore(squad): flush session state for v1.7.0 ship + workflow simplification` commit (5c92b75, HEAD on main after v1.7.0 release) triggered release-please to open **PR #1113** proposing a no-op v1.7.1 release. Martin closed it with comment:

> "Closing as no-op release. The chore(squad) flush commit doesn't ship user-visible changes, so a v1.7.1 bump for it alone is noise. Next real fix/feat commit will trigger a proper release PR that includes the squad chore in CHANGELOG."

**Question:** Should we prevent this pattern, or accept it?

---

### Investigation: Why Did `chore(squad)` Trigger a Release?

#### 1. Release-Please Config Examination
**File:** `release-please-config.json`
```json
{
  "release-type": "simple",
  "changelog-sections": [
    { "type": "feat", "section": "Features" },
    { "type": "fix", "section": "Fixes" },
    { "type": "perf", "section": "Performance" },
    { "type": "deps", "section": "Dependencies" },
    { "type": "docs", "section": "Documentation" },
    { "type": "chore", "section": "Chores" },
    { "type": "ci", "section": "CI" },
    { "type": "test", "section": "Tests" },
    { "type": "refactor", "section": "Refactors" }
  ]
}
```

**Key insight:** `"type": "chore"` is present in `changelog-sections`. However, **changelog-sections controls rendering only, not version bumping.**

#### 2. Release-Please Bump Logic
Per googleapis/release-please documentation and observed behavior:
- `feat:` → MINOR version bump
- `fix:`, `perf:` → PATCH version bump
- `docs:`, `test:`, `ci:`, `chore:`, `refactor:`, etc. → **NO version bump UNLESS BREAKING CHANGE detected**

**So why did #1113 propose v1.7.1?**

#### 3. Release-Please Behavior: All Commits Since Last Tag
Release-Please's core logic in `simple` release-type mode:
1. Find the last release tag (`v1.7.0` = d3181a6, published 2026-05-13 17:49:46)
2. Scan commits from HEAD backward to the tag
3. If ANY commit matches a bump-triggering type (`feat`, `fix`, `perf`), open a release PR
4. If only non-bump types exist (`docs`, `test`, `ci`, `chore`, `refactor`, etc.), some versions of release-please **still open a release PR** to update CHANGELOG and `.release-please-manifest.json`, even without bumping the version

**Evidence from PR #1113:**
- Diff files: `.release-please-manifest.json`, `AzureAnalyzer.psd1`, `CHANGELOG.md`
- CHANGELOG addition: "### Chores" section with the commit
- **Version in AzureAnalyzer.psd1: UNCHANGED at `1.7.0`**  ← This is the key!
- **Version in .release-please-manifest.json: UNCHANGED at `1.7.0`**

**Actual behavior:** Release-please **opened an empty release PR** (no version bump, but CHANGELOG + manifest update) because:
- A commit existed between the tag and HEAD
- The commit type matched a configured `changelog-section` (`chore`)
- Even though it doesn't bump, release-please still offers to "prepare the release" (i.e., update CHANGELOG and metadata)
- This is release-please's default behavior in `simple` mode when commits exist post-tag — it doesn't know whether to ignore them until it runs, so it opens a draft PR for review

---

### Release-Please Configuration Details
**Workflow:** `.github/workflows/release.yml` lines 47–53
```yaml
- name: Run release-please (manifest mode)
  uses: googleapis/release-please-action@5c625bfb5d1ff62eadeeb3772007f7f66fdcf071 # v4
  with:
    token: ${{ steps.app-token.outputs.token }}
    command: manifest
    config-file: release-please-config.json
    manifest-file: .release-please-manifest.json
```

**Default behavior** for v4 action + `simple` release-type + manifest mode:
- Opens a release PR for any commits post-tag that match `changelog-sections` types
- Does NOT filter out `chore`/`docs`/`test` commits from triggering a release candidate PR
- Humans manually close no-op PRs, or configuration tightens the filter

---

### Root Cause Analysis
1. **Is `chore` in `changelog-sections` causing the bump?** No — changelog-sections is rendering only.
2. **Is release-please v4 opening a draft release PR for any commit post-tag?** Yes.
3. **Does release-please v4 have a built-in "skip chore-only commits" flag?** Partially — there are no out-of-the-box filters, but config can exclude commit types via advanced settings.

---

### Three Policy Options

#### **(A) ACCEPT: Manually close no-op release PRs**
**Rationale:**
- Minimal config change; release-please v4 continues to work as-is
- Rare cost: close a PR every time `chore(squad)` lands post-release (happens ~1-2x per sprint during squad state flushes)
- Benefit: zero risk of accidentally filtering out a real `fix` or `feat` buried in a `chore` commit message

**Config:** No change.  
**Toil:** ~2 min per occurrence (close PR + verify next real release PR lands correctly).  
**Risk:** Low.

**Recommendation:** Choose (A) if the team tolerates occasional manual PR closes.

---

#### **(B) FILTER: Exclude non-bump types from triggering release PRs**
**Rationale:**
- Automatic: release-please doesn't open a PR unless a `feat`/`fix`/`perf` commit exists
- Reduces toil: no manual PR closes

**Config change:** Add bump-rule filter to `release-please-config.json`:
```json
{
  "release-type": "simple",
  "bump-only-if": ["feat", "fix", "perf"],
  "changelog-sections": [
    { "type": "feat", "section": "Features" },
    { "type": "fix", "section": "Fixes" },
    { "type": "perf", "section": "Performance" },
    { "type": "docs", "section": "Documentation" },
    { "type": "chore", "section": "Chores" },
    { "type": "ci", "section": "CI" },
    { "type": "test", "section": "Tests" },
    { "type": "refactor", "section": "Refactars" }
  ]
}
```

**Toil:** 0 — fully automated.  
**Risk:** Medium. If a commit has multiple footers or interleaved types (e.g., "feat: X\n\nBREAKING CHANGE"), filter might miss it. Requires validation that all real features/fixes continue to trigger releases (test via force-push + workflow_dispatch trigger).

**Recommendation:** Choose (B) if the team wants zero manual toil AND is willing to validate the filter in a test PR.

---

#### **(C) TARGETED: Change `chore(squad):` prefix to bypass release-please**
**Rationale:**
- Surgical: only `squad:` prefixed commits skip release-please
- All other `chore:`, `docs:`, `test:` commits still trigger release PRs (good for visibility)
- Squad state flushes become completely invisible to release machinery

**Config change:** Update `chore(squad):` commits to use a non-conventional prefix like `squad:` instead.

**Example:**
```bash
git commit -m "squad: flush session state for v1.7.0 ship + workflow simplification (#1112)"
```

Then add to `.squad/ceremonies.md` or `.github/CONTRIBUTING.md`:
> Commits with `squad:` prefix are internal state synchronization and do not trigger release-please. These are used for session state flushes, decision board updates, and other cross-repo coordination.

**Toil:** 0 — fully automated once convention adopted.  
**Risk:** Low. The `squad:` prefix is already in the spirit of squad-internal work; this just extends it.

**Recommendation:** Choose (C) if the team wants the most semantic solution (squad work stays squad-internal, doesn't pollute releases).

---

### Recommendation: **(A) ACCEPT**

**Why (A)?**

1. **Simplicity:** No config risk, no convention changes.
2. **Frequency:** `chore(squad)` flushes are infrequent (1-2 per major release cycle). Closing one PR every 2-3 weeks is negligible toil.
3. **Safety:** (B) risks filtering legitimate commits; (C) requires convention adoption across team.
4. **Reversibility:** If toil increases, easy to switch to (B) or (C) later.

**Manual Process (when PR #1113-style happens again):**
1. Observe PR from release-please-action[bot] proposing an empty bump
2. Scan CHANGELOG additions: if only `chore`/`docs`/`test`, it's a no-op
3. Leave a comment explaining (for the record)
4. Close the PR
5. Next real `fix` or `feat` commit will trigger a legitimate release PR with the chore included in CHANGELOG

**Cost:** ~1 min per occurrence. Done.
