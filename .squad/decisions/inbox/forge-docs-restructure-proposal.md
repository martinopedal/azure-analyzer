# Docs Restructure Proposal: Progressive Disclosure, Crystal Clarity, Expandable Depth

**Proposal ID:** forge-docs-restructure

**Date:** 2026-04-23 (Refined 2026-04-23)

**Charter:** Platform Automation & DevOps Engineer

**Reviewer:** Martin

**Principles:** Progressive disclosure. Crisp purpose statements. Collapsible advanced/reference sections. Dual-audience design where applicable. No em-dashes.

---

## Executive Summary

Azure-analyzer documentation needs structural clarity, progressive disclosure (simple operations visible, advanced depth in collapsibles), and crisp purpose contracts per page. Current docs are oversized, mix levels of detail without visual hierarchy, and bury the "what is this" / "how do I run it" contracts under feature lists and references.

This proposal reorganizes docs into user/operator/contributor/architecture segments (flattened nesting), applies progressive disclosure throughout (collapsible `<details>` blocks for advanced/reference), defines crisp contracts for README and tool pages, adds an ETL reference page with field mappings, adds a parameter reference page, and establishes visibility policies per section. No file moves yet. Proposal-only.

---

## 1. Current State Problems (Triage)

### Clarity and Progressive Disclosure
1. **README oversized (130 lines)**: Mixes landing pitch, quickstart, feature tour, schema reference, tool list, permissions summary. User's eye lands on feature matrix before "how do I run it".
2. **Tool pages lack crisp contracts**: Current tool docs buried under permissions details; no "at-a-glance" what/why/how structure.
3. **ETL and field mapping scattered**: Schema 2.2 field mappings, v1-to-v3 flow, and normalizer specifics spread across README, ARCHITECTURE.md, CHANGELOG.md, and per-tool pages. No dedicated reference.
4. **Parameter reference missing**: Invoke-AzureAnalyzer.ps1 has 20+ params. No single-page listing with defaults, examples, and "when to use" guidance grouped by consumer/advanced.

### Structure and Navigation
5. **No docs index**: Landing in `docs/` offers no TOC. Users must guess consumer vs. contributor vs. operators.
6. **Orphaned stubs with deadlines**: 7 files redirect; tracked in `.squad/stub-deadlines.json`. Maintenance overhead.
7. **Dual audience (consumer + contributor) mixed**: Same pages try to serve "how to use" and "how to extend" without clear audience switching.

## 3. Progressive Disclosure: Visibility Policy Per Page

Every page applies this contract:
- **First 10 seconds of scrolling (no collapse)**: Purpose, inputs, outputs, minimal example.
- **Inside `<details>` (default-closed)**: Advanced params, full ETL mapping, edge cases, gotchas.
- **Nesting rule**: Max 2 levels of `<details>` nesting. Any deeper breaks readability.
- **Audience split**: Use side-by-side collapsibles ("For consumers" / "For maintainers") when content diverges.

### ETL Pipeline Reference (`docs/reference/etl-pipeline.md`)

**Visible top section**:
```markdown
# ETL Pipeline: From Raw Tools to Unified Schema

**One sentence**: Wrappers collect raw tool output (v1 envelope), normalizers transform to Schema 2.2 FindingRow (v2), and the orchestrator deduplicates into entity-centric records (v3 entities).

**Flow**:
```
Raw Tool Output
  |
  v
Wrapper (Invoke-*.ps1) -> v1 Envelope (standard JSON shape with raw findings)
  |
  v
Normalizer (Normalize-*.ps1) -> New-FindingRow -> v2 FindingRow (standard 32-field schema)
  |
  v
Orchestrator -> EntityStore -> v3 Entities (deduplicated, cross-tool entity refs)
  |
  v
Report (HTML/MD) + findings.json + entities.json
```

**What each layer does**:
1. **Wrapper**: Runs the tool, captures stdout/stderr, normalizes the JSON envelope. All output validated against v1 contract (guid, source, compliant, detail, findings array).
2. **Normalizer**: Maps raw fields to Schema 2.2 (severity, frameworks, pillar, impact, effort, evidence URIs, remediation snippets, etc.). Input: v1, output: v2 FindingRow.
3. **Entity Store**: Groups findings by entity (canonical IDs), deduplicates, merges metadata. Output: entities.json (deduplicated) + results.json (detailed findings).
```

## 4. Updated README Contract (Specific)

**Top-level visible section (NO SCROLLING for first 50 lines)**:
1. Badges (CI, CodeQL, License)
2. One-sentence pitch (what is azure-analyzer)
3. Three-bullet list (what you get: findings JSON, HTML+MD reports, entity store)
4. Single command example (how to run)
5. Link to docs/getting-started/ for the real quickstart

**Everything else**: Collapsed `<details>` blocks (feature matrix, tool catalog summary, permissions, contributor links).

**Target**: User's eye lands on "WHAT IS THIS" and "HOW DO I RUN IT" in under 10 seconds; can expand for more, but doesn't need to.

## 6. Collapsible Best Practices (Accessibility)

1. **Keyboard-accessible**: GitHub's `<details><summary>` is native, keyboard-navigable (Tab, Enter, Shift+Tab).
2. **Screen-reader friendly**: Summary text reads as button. Screen readers announce expanded/collapsed state.
3. **Nesting rule**: Max 2 levels. Any deeper breaks readability and loses users.
4. **Summary text**: Always a complete, actionable phrase. Bad: "Details". Good: "Full parameter list for azqr".
5. **Default state**: `<details open>` for critical info (quickstart, common params). `<details>` (closed) for advanced/reference.
