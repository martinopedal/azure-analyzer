- 2026-04-22 — Maester `ConvertTo-MtMaesterResult` is the canonical structure-of-truth for what the wrapper SHOULD capture. Key fields the wrapper currently drops: `Tag` (Block.Tag + Test.Tag — drives compliance-framework badges like EIDSCA/CIS/MITRE/eIDAS2/NIST), `ResultDetail.TestDescription`/`TestResult`/`TestRemediation` (all markdown — must be rendered as HTML, not flattened), `HelpUrl` parsed from `See https://...` in test name, `Investigate` and `Error` result states (both squashed to Compliant=true today), `Duration`, `ScriptBlock`/`ScriptBlockFile`, Maester module version. Severity comes from `ResultDetail.Severity` (set by `Add-MtTestResultDetail -Severity` or `Get-MtMaesterConfigTestSetting`), with values Critical|High|Medium|Moderate|Low|Info — our existing word-boundary regex in `Normalize-Maester.ps1` already handles `moderate→Medium` correctly.
- 2026-04-22 — Maester native HTML report (4 MB single-file React-style bundle in `powershell/assets/ReportTemplate.html`) IA: branded header → 6-tile counter strip (Total/Passed/Failed/Investigate/Skipped/NotRun, doubles as filter chips) → per-block summary table → expandable test rows. Each row split as `TestId: Title` (e.g. `EIDSCA.AF01: Authentication Method – FIDO2 – State`) with severity pill, tag chips, and on-expand Description+Evidence+Remediation panels (all markdown). Includes "About this run" panel with `MgContext`, `SystemInfo`, `PesterConfiguration` for auditor reproducibility. Decision dropped at `.squad/decisions/inbox/iris-maester-ui-patterns.md` for Sentinel.
- 2026-04-22 — Kubescape v2 JSON (`--format json --format-version v2`) carries `summaryDetails.score` (cluster posture 0–100, hero KPI), `summaryDetails.complianceScore`, `summaryDetails.frameworks[]` (per-framework score), `summaryDetails.statusCounters` (passed/failed/skipped/**excluded**/**irrelevant** — 5 buckets, not 3), and per-control `scoreFactor` (1–10 → our Critical/High/Medium/Low bands at ≥9 / ≥7 / ≥4 / else). Per-resource details live in `results[].controls[].rules[]` with `paths` (failed JSONPath) and `fixPaths` (assisted-remediation YAML). Frameworks observed: `NSA`, `MITRE`, `cis-v1.10.0`, `cis-v1.23-t1.0.1`, `cis-aks-t1.2.0`, `cis-eks-t1.7.0`, `ArmoBest`, `AllControls`, `WorkloadScan`, `DevOpsBest`, `soc2`, `security`, `ClusterScan`. Our wrapper drops ALL of: framework membership, control description, real remediation text, fixPaths, scoreFactor, posture score, statusCounters, and per-resource granularity (we collapse N affected pods into 1 cluster-level finding). Recommendation to Sentinel: unify `<framework-chip>` component across Maester+Kubescape (same colors, framework-keyed not tool-keyed, so CIS is always amber regardless of source); keep posture score and severity enum as separate KPIs (different questions); emit one FindingRow per `(controlID, resourceID)` pair with cluster as `EntityId` and K8s coordinates in `SubResourceId`. Decision dropped at `.squad/decisions/inbox/iris-kubescape-ui-patterns.md`.
- 2026-04-22 — ETL gap remediation pattern (codified in both Maester and Kubescape inbox drops under `## ETL gaps end-to-end (revised)`): every dropped tool field needs a fix at all 5 layers — wrapper → normalizer → schema → entity store → report. Schema v2.1 already provisions `Frameworks: object[]`, `Controls: string[]`, `RuleId`, `Confidence`, `MissingDimensions`, `LearnMoreUrl`, `Remediation`, `Category` on `FindingRow` (`Schema.ps1:206–231`); most "dropped" fields just need wrappers/normalizers to populate them. Three shared schema additions needed for both tools (land once): (1) `New-FindingRow -Properties [object]` open container (bump to v2.2, additive, back-compat); (2) entity-store envelope v3.1 → v3.2 with `RunContexts: object[]` sidecar for per-tool/per-run scan metadata (posture scores, status counters, MgContext, tool versions); (3) shared `modules\shared\Frameworks.ps1` with `Get-FrameworkFromTag` classifier so EIDSCA/CIS/MITRE/NSA tags from any tool render with identical chip styling. First-class vs. container rule of thumb: cross-tool concepts and queryable identifiers go first-class (Frameworks, RuleId, LearnMoreUrl, Remediation); long markdown bodies, tool-specific numerics, and audit payloads go in `Properties`; per-tool/per-run metadata (tool version, scan context, cluster posture score) goes in `RunContexts` (envelope sidecar, not on every row). Per-resource granularity (Kubescape SubResourceId) deferred from EntityType promotion until we design cluster→namespace→workload Edges.
